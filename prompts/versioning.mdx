Every change to your prompts is tracked with semantic versioning. This gives you complete visibility into how your prompts evolve over time and makes it easy to rollback to previous versions if needed.

## Version Structure

Helicone uses semantic versioning with major and minor version numbers:

- **Major versions** (e.g., v1.0, v2.0) - Significant prompt changes or structural updates
- **Minor versions** (e.g., v1.1, v1.2) - Incremental improvements within a major version

Each version includes:
- **Version ID** - Unique identifier for the version
- **Commit Message** - Description of what changed
- **Created At** - Timestamp of when the version was created
- **Model** - The LLM model configured for this version
- **Environments** - Which environments this version is deployed to

## Creating New Versions

You can create new versions directly from the Helicone dashboard:

<Steps>
  <Step title="Edit Your Prompt">
    Make changes to your prompt in the playground - modify messages, adjust parameters, or update variables.
  </Step>

  <Step title="Choose Version Type">
    Decide whether this is a major or minor version change:
    
    - **Minor version** - For tweaks, refinements, or small improvements
    - **Major version** - For significant changes that might affect behavior
  </Step>

  <Step title="Add Commit Message">
    Write a clear commit message describing what changed and why. This helps your team understand the evolution of your prompts.

    <CodeGroup>
      ```text Good Commit Messages
      "Improved response tone to be more empathetic"
      "Added support for multi-language responses"
      "Fixed variable substitution in tool schema"
      "Optimized token usage by simplifying system prompt"
      ```

      ```text Avoid
      "Updated prompt"
      "Changes"
      "V2"
      ```
    </CodeGroup>
  </Step>

  <Step title="Save Version">
    Save the new version. It's immediately available for testing but not deployed to any environment until you explicitly deploy it.
  </Step>
</Steps>

## Version Selection at Runtime

When making API calls, you can specify which version to use in several ways:

### Using Environments (Recommended)

Deploy versions to specific environments and reference them in your code:

<CodeGroup>
  ```typescript TypeScript
  import { OpenAI } from "openai";
  import { HeliconeChatCreateParams } from "@helicone/helpers";

  const openai = new OpenAI({
    baseURL: "https://ai-gateway.helicone.ai",
    apiKey: process.env.HELICONE_API_KEY,
  });

  // Use production version
  const response = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    prompt_id: "abc123",
    environment: "production",
    inputs: { name: "Alice" }
  } as HeliconeChatCreateParams);
  ```

  ```python Python
  import openai
  import os

  client = openai.OpenAI(
      base_url="https://ai-gateway.helicone.ai",
      api_key=os.environ.get("HELICONE_API_KEY")
  )

  # Use staging version
  response = client.chat.completions.create(
      model="gpt-4o-mini",
      prompt_id="abc123",
      environment="staging",
      inputs={"name": "Alice"}
  )
  ```
</CodeGroup>

### Using Specific Version IDs

For testing or debugging, you can use a specific version directly:

<CodeGroup>
  ```typescript TypeScript
  const response = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    prompt_id: "abc123",
    version_id: "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    inputs: { name: "Alice" }
  } as HeliconeChatCreateParams);
  ```

  ```python Python
  response = client.chat.completions.create(
      model="gpt-4o-mini",
      prompt_id="abc123",
      version_id="xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
      inputs={"name": "Alice"}
  )
  ```
</CodeGroup>

### Default Behavior

If you don't specify an environment or version ID, the production version is used:

<CodeGroup>
  ```typescript TypeScript
  // Uses production version by default
  const response = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    prompt_id: "abc123",
    inputs: { name: "Alice" }
  } as HeliconeChatCreateParams);
  ```

  ```python Python
  # Uses production version by default
  response = client.chat.completions.create(
      model="gpt-4o-mini",
      prompt_id="abc123",
      inputs={"name": "Alice"}
  )
  ```
</CodeGroup>

<Note>
  **Priority**: If both `environment` and `version_id` are provided, the environment takes precedence.
</Note>

## API Reference

### Get Prompt Version

Retrieve metadata for a specific version:

<CodeGroup>
  ```typescript TypeScript
  const response = await fetch(
    "https://api.helicone.ai/v1/prompt-2025/query/version",
    {
      method: "POST",
      headers: {
        "Authorization": `Bearer ${HELICONE_API_KEY}`,
        "Content-Type": "application/json"
      },
      body: JSON.stringify({
        promptVersionId: "version-uuid"
      })
    }
  );

  const result = await response.json();
  // Returns: { data: Prompt2025Version, error?: string }
  ```

  ```python Python
  import requests
  import os

  response = requests.post(
      "https://api.helicone.ai/v1/prompt-2025/query/version",
      headers={
          "Authorization": f"Bearer {os.environ.get('HELICONE_API_KEY')}",
          "Content-Type": "application/json"
      },
      json={
          "promptVersionId": "version-uuid"
      }
  )

  result = response.json()
  # Returns: { "data": Prompt2025Version, "error"?: str }
  ```

  ```bash cURL
  curl -X POST https://api.helicone.ai/v1/prompt-2025/query/version \
    -H "Authorization: Bearer $HELICONE_API_KEY" \
    -H "Content-Type: application/json" \
    -d '{
      "promptVersionId": "version-uuid"
    }'
  ```
</CodeGroup>

### Response Type

All version endpoints return a `Prompt2025Version` object:

```typescript
interface Prompt2025Version {
  id: string;                    // Version UUID
  model: string;                 // Model name (e.g., "gpt-4o-mini")
  prompt_id: string;             // Parent prompt ID
  major_version: number;         // Major version number
  minor_version: number;         // Minor version number
  commit_message: string;        // Version commit message
  environments?: string[];       // Deployed environments
  created_at: string;           // ISO timestamp
  s3_url?: string;              // S3 URL for prompt body
  prompt_body?: {               // Only included when explicitly requested
    model?: string;
    messages?: Array<{...}>;
    temperature?: number;
    max_tokens?: number;
    tools?: Array<{...}>;
    // ... other OpenAI parameters
  };
}
```

## Best Practices

<AccordionGroup>
  <Accordion title="Use Meaningful Commit Messages">
    Write clear, descriptive commit messages that explain:
    - What changed
    - Why it changed
    - Any important context

    This makes it easy for your team to understand prompt evolution over time.
  </Accordion>

  <Accordion title="Test Before Deploying">
    Create new versions and test them thoroughly before deploying to production:
    1. Create new version
    2. Test with the version_id parameter
    3. Deploy to staging environment
    4. Run integration tests
    5. Promote to production
  </Accordion>

  <Accordion title="Use Major Versions for Breaking Changes">
    Create a new major version when:
    - Changing the response structure significantly
    - Modifying variable names or types
    - Restructuring the conversation flow
    - Changing model providers

    Use minor versions for:
    - Tweaking temperature or other parameters
    - Refining the prompt text
    - Adding optional variables
  </Accordion>

  <Accordion title="Keep Version History Clean">
    Don't create unnecessary versions. Group related changes together:
    - Make multiple edits in the playground
    - Test thoroughly
    - Create one version with all related changes
  </Accordion>
</AccordionGroup>
